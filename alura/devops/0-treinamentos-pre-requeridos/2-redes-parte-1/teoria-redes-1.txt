Aula 01: A rede com IP, ping e Traceroute -------

    Teoria:

        Introdução:

            Veremos neste curso...

                - Conexão dos cabos
                - Equipamentos de hub
                - Switch
                - Roteadores
                - Servidores
                - DNS e DHCP
                - IPV4
                - Segurança da informação dos equipamentos (Básico)

                Iremos montar um projeto usando um software de simulação de rede e faremos testes de acesso e conexão.

        Definindo a rede:

            Quando fazemos uma conexão com o Google, por exemplo, nossa requisição passa por diversos pontos, regiões, equipamentos e computadores intermediários antes de chegar de fato 
            no google. Da mesma forma acontece com a resposta do Google só que fazendo o caminho inverso.

            As diversas máquina e equipamentos interconectados em diferentes pontos, capazes de transportar informações é o que caracterizam a nossa rede. Elas existem em diferentes 
            tamanhos, desde um ambiente doméstico, em que temos dois computadores e uma impressora, até grandes corporações com centenas de usuários, máquinas potentes com servidores, até
            a mais famosa, a internet.

        Ping:

            No mundo da internet a conexão e entrega de dados é parecida com o mundo real, onde enviamos uma carta que contém em suas informações quem é o destinatário e qual é o endereço. 
            O processo de identificação das máquinas é chamado de endereçamento IP.

            Para ver como ele funciona, abriremos o terminal do S.O que estamos trabalhando e utilizamos o comando ipconfig (ifconfig no Mac).

            A informação que segue o label Endereço IPV4 é o IP da nossa máquina.

            Caso não consigamos uma conexão com determinada máquina na rede, poderemos utilizar o protocolo ping para verificar se ela está ativa ou apta para conexão.

            Podemos utilizar o protocolo ping para verificar se uma máquina está ativa ou apta para conexão. No caso, quando queremos testar a conexão como uma máquina distante, podemos 
            pingar o endereço, no caso do google...www.google.com.br o que nos retornará o seu número IP. Isto é a tradução da URL para o endereçamento IP. O responsável por fazer esta 
            tradução ou mapeamento é o chamado servidor DNS.

                Para realizar o teste, digitamos no nosso terminal o seguinte comando: ping www.google.com.br

                A resposta que obtivemos foi a seguinte:

                    C:\Users\marlon.narcizo>ping www.google.com.br

                    Disparando www.google.com.br [172.217.29.3] com 32 bytes de dados:
                    Resposta de 172.217.29.3: bytes=32 tempo=33ms TTL=52
                    Resposta de 172.217.29.3: bytes=32 tempo=39ms TTL=52
                    Resposta de 172.217.29.3: bytes=32 tempo=35ms TTL=52
                    Resposta de 172.217.29.3: bytes=32 tempo=41ms TTL=52

                    Estatísticas do Ping para 172.217.29.3:
                        Pacotes: Enviados = 4, Recebidos = 4, Perdidos = 0 (0% de
                                perda),
                    Aproximar um número redondo de vezes em milissegundos:
                        Mínimo = 33ms, Máximo = 41ms, Média = 37ms


            Podemos notar a tradução da URL para o endereçamento IP do Google.

            Dentro da informação do ping existe um protocolo chamado ICMP (Internet Control Message Protocol), este protocolo trabalha de forma semelhante a um telefonema. É como se 
            pegássemos um telefone e ligássemos para a máquina do Google, na esperança de que alguem atenda. Dentro da informação enviada, falaremos algo que será representado pelos 32 
            bytes que aparecem no retorno.

            A máquina do Google responde, podemos ver pelo IP dela e pelo tempo que a informação levou do nosso computador para passar pela máquina do Google e retornar. Isto é chamado de
            tempo de Ida e Volta ou RTT (Round Trip Time). Para que seja realizada esta conexão, precisamos passar por diversas máquinas intermediárias, que podem estar configuradas de 
            diversas forma que caracterizam o loop, porque existem várias interconexões. Se a nossa informação enviada ficar trafegando eternamente no loop, consumindo recursos, travaria 
            tudo.

            Por isso que cada pacote de informação tem um tempo de vida útil e a cada passagem entre essas máquinas, ele irá decrementar em uma unidade representado pelo índice 
            TTL (Time to live).

            No nosso exemplo, quando a informação passar por 52 máquinas, o valor será zerado e extinguida.

            Podemos ver também que enviamos 4 pacotes para a máquina do Google e todos eles foram respondidos, isto significa que a conexão está perfeita.

        
        Traceroute:

            O Traceroute através do ICMP faz uma verificação para saber quais são as máquina intermediárias que participam do processo.
                Obs: No Linux considerar traceroute -n.

            Para testar o tracerout utilizamos o comando: tracert -d www.google.com
                Obs: O -d (-n no Linux) para que ele não faça a tradução do DNS.

            A minha saída foi:

                C:\Users\nesso>tracert -d www.google.com

                Rastreando a rota para www.google.com [2800:3f0:4001:810::2004]
                com no máximo 30 saltos:

                1    <1 ms    <1 ms    <1 ms  fe80::7ac1:a7ff:fe1a:3102
                2     *        *        *     Esgotado o tempo limite do pedido.
                3     *        *        *     Esgotado o tempo limite do pedido.
                4    62 ms    63 ms    63 ms  2001:4860:1:1::f74
                5    74 ms    62 ms    63 ms  2800:3f0:800e::1
                6    63 ms    64 ms    63 ms  2001:4860:0:1::1eba
                7     *       74 ms    74 ms  2800:3f0:4001:810::2004

                Rastreamento concluído.

            Podemos observar que foi feita a mesma tradução, por meio do servidor DNS, que no protocolo ping. Mas as informações retornadas foram diferentes.

            Na primeira coluna temos a identificação da máquina, a sequencia praticamente por onde passou a informação da minha máquina. Assim como ela passará por outras 
            máquinas.

            Observamos também que, temos os 3 intervalos de tempo que são mostrados nas outras máquinas. Eles são os mesmos do ping se referem ao processo de ida do cliente para o servidor
            e a volta do servidor para o cliente.

            Na vida real temos um caminho em mente que vamos percorrer pra chegar em casa, porém este caminha pode estar congestionado, e assim como na vida real teremos que escolher outro
            caminho. Desta mesma forma pode acontecer na rede, podemos enviar um pacote com informações, mas pode ser que seja necessário que este pacote encontre outro caminho para chegar
            no servidor. Para identificar se será necessário enviar numa rota diferente, três pacotes diferentes em intervalos diferentes são enviados para ver se alguma outra máquina
            poderia atuar como ponto de parada.

            Na primeira coluna podemos perceber que querm recebeu os três pacotes foi o mesmo IP, ou seja, a primeira coluna se refere a mesma rota. Porém a máquina 2 e 3 não retornaram
            nenhuma informação de tempo. Elas não responderam por dois motivos possíveis. O primeiro é porque o ICMP pode estar desabilitado para evitar sobrecarga, o segundo pode ser questão
            de segurança, pois um dos primeiros testes de invasão de rede é um teste de conectividade como o que estamos fazendo. Ele começa por uma verificação de qual é o sistema que a 
            empresa está trabalhando, quais as possíveis portas que estão abertas para conseguir acesso. Mas no nosso caso a máquina está funcionando, pois passaram a informação para a 
            máquina 4 que continuou passando para a próxima. Então chegou na 7, que é o servidor Google.

                 7     *       74 ms    74 ms  2800:3f0:4001:810::2004

        O que vimos nessa aula

        - O que é Rede de computadores.
        - O comando ping para teste de conectividade.
        - O protocolo ICMP utilizado
        - O (Time to Live) TTL OU tempo de vida útil do pacote.
        - O comando traceroute para verificar a rota que meu pacote está passando na rede.

    
    Questões aula 01:

        1 - O que caracteriza uma rede?

            Selecione uma alternativa

            R: Máquinas que estão interligadas, passando informações em diversos pontos

            De acordo com o site do Wikipedia, a melhor definição de redes de computadores seria: “Uma rede de computadores é formada por um conjunto de máquinas eletrônicas com 
            processadores capazes de trocar informações e partilhar recursos”. Ou seja, pelo fato das máquinas estarem interligadas e serem capazes de trocar informações, caracterizaria 
            uma rede.


        2 - Qual protocolo está dentro do ping?

            Selecione uma alternativa

            R: ICMP

            Dentro da ferramenta administrativa do ping temos o protocolo ICMP, sendo ele o responsável por mandar uma requisição (Echo Request) para máquina remota e esperar um retorno 
            dessa máquina remota (Echo Reply).

        
        3 - Caso o tempo indicado no teste do ping esteja mais elevado em relação ao valor que normalmente apresenta, o que isso quer dizer?

            Selecione uma alternativa

            R: Isso significa que podemos ter um problema tanto na comunicação de ida ou da volta

            Quando temos um alto valor no índice do tempo do teste do ping, isso significa que podemos ter um problema em nossa comunicação. O protocolo ICMP que está dentro do ping manda
            um Echo request e aguarda um retorno de resposta da máquina destino (Echo Reply). Dessa forma, o tempo elevado pode indicar um problema na comunicação que pode estar tanto em 
            um trecho como em outro.

        4 - O que seria o TTL que aparece no teste do ping?

            Selecione uma alternativa

            R: TTL indica por quantas máquinas (hops) minha informação pode passar antes de ser extinta

            O TTL seria uma informação dentro do pacote do IP que informa qual é a máxima quantidade de hops que minha informação pode passar antes de ser descartada. É a quantidade de 
            máquinas que ela vai poder passar no caminho.

        
        5 - Quando digitamos ping www.google.com no console aparece um número, o que seria esse número?

            Selecione uma alternativa

            R: Endereço de IP

            Ao digitarmos www.google.com, ocorre uma tradução entre o nome e o endereço IP da máquina do google que estamos acessando. As máquinas para serem identificadas na rede devem 
            possuir um endereçamento IP.

        
        6 - Qual ferramenta administrativa eu uso para testar conectividade?

            Selecione uma alternativa

            R: ping

            Dentro do ping temos o protocolo ICMP responsável por fazer a requisição até uma máquina remota (Echo Request) e esperar a resposta (Echo Reply). Através desse protocolo ICMP, 
            podemos verificar se temos uma resposta da máquina remota, sabendo assim se temos conectividade.

            O traceroute seria usado para verificar a rota que minha informação percorreu até chegar o destino.

            O ipconfig seria usado para ver as configurações de IP de minha máquina e o nslookup seria para verificarmos recursos mais avançados de problemas que podemos ter entre a url e 
            o endereço IP.
            
        
        7 - Qual o principal uso do traceroute?

            Selecione uma alternativa

            R: Verificar a rota que a minha informação está passando na rede

            A principal funcionalidade do traceroute é verificar a rota que a minha informação levou para chegar até a máquina de destino. Isso porque, em redes de computadores temos o 
            que chamamos de rede não determinística, ou seja, não necessariamente um pacote de informação vai ser transferido pela mesma rota do anterior com o mesmo intervalo de tempo. 
            Isto se deve a muitos fatores, por exemplo, uma máquina que pode estar congestionada ou um problema no link de comunicação, etc.

            Pense como se a rede fosse uma cidade com várias ruas, nem sempre pegaremos a mesma rota para chegar até a nossa casa ou ir para o trabalho, depende de muitos fatores, por 
            exemplo, trânsito, obras e outros aspectos. Nas redes de computadores não é tão diferente. :)

        
        8 - Quando eu digito tracert e vemos que uma máquina retornou a informação (*), o que isso quer dizer?

            Selecione uma alternativa

            R: Provavelmente o administrador da máquina desabilitou a resposta ao nosso chamado, para evitar um grande volume de tráfego na máquina bem como por questões de segurança.

            Quando nós temos uma máquina que retornou (*) e passou a informação para uma próxima máquina, isso provavelmente indica que o administrador dessa máquina desabilitou a 
            resposta ao nosso chamado. O que acontece seria que esse tipo de teste pode ser interpretado como uma tentativa de “scanear” possíveis portas abertas e vulnerabilidades que 
            possam existir, caso seja usado por um usuário malicioso, pode ser usada como uma forma de reconhecimento da rede dessa possível vítima para que assim possa explorar possíveis
            falhas.

Aula 03: Cabos de conexão -------

    Teoria:

        O cabo de rede que compramos, possui 8 fios por dentro, e suas cores representam um padrão que veremos a seguir.

            (T568A)

            1 - "Branco do verde"
            2 - Verde
            3 - "Branco do laranja"
            4 - Azul
            5 - "Branco do azul"
            6 - Laranja
            7 - "Branco do marrom"
            8 - Marrom

        O conector de plástico no qual encaixamos e crimpamos (Ato de fixar com alicate os fios no conector) os fios menores, é chamado de RJ-45.

        Como já vimos, temos esta combinação de cores acima, esta sequencia de cores e este padrão designado para tal sequencia foi criado pela TIA (Telecomunication industry
        association). Eles definiram o padrão de cores que devia ser seguido. Agora vamos analisar as conexões que fizemos até agora no curso.

        Quando os fios são crimpados com a sequencia de cores do padrão T568A o cabo é chamado de cabo direto.

            A placa de rede transmite nas posições 1 e 2:

                1 - "Branco do verde"
                2 - Verde

            E receberá nas posições 3 e 6:

                3 - "Branco do laranja"
                6 - Laranja


            Já o hub por sua vez, transmitirá nas posições:

                3 - "Branco do laranja"
                6 - Laranja

            E receberá nas posições:

                1 - "Branco do verde"
                2 - Verde

            Praticamente é como se a placa do hub fosse ao contrário.

        Porém se utilizarmos o mesmo cabo em dois computadores que por convenção terão suas placas "no mesmo sentido", a transmissão e recepção dos dois serão tentadas a partir
        dos mesmos fios, o que ocasionará a colisão. Pois estão transmitindo pelo mesmo fio (1 - "Branco do verde", 2 - Verde) e tentando receber pelos mesmos fios 
        (3 - "Branco do laranja", 6 - Laranja).

        Pensando nisto, a TIA criou um outro padrão, chamado T568B.

            (T568B)

            1 - "Branco do laranja"
            2 - Laranja
            3 - "Branco do verde"
            4 - Azul
            5 - "Branco do azul"
            6 - Verde
            7 - "Branco do marrom"
            8 - Marrom

        Ou seja, uma ponta com o T568A e a outra ponta com o T568B, será o ideal, pois o computador 1 transmite e na posição 3 o computador 2 recebe. E também a posição 2
        transmitirá e a posição 6 receberá.

            Então percebemos que, nos computadores, o padrão é transmitir nas posições 1 e 2, e receber nas posições 3 e 6.
            Se invertermos então os cabos Branco do verde e verde, e Branco do laranja e Laranja apenas em uma ponta dos cabos, a comunição funcionará, pois aconteceu um
            cruzamento interno dos sinais de uma forma que eles não colidam. Por causa deste cruzamento este tipo de cabo recebeu o nome de cabo cruzado ou crossover.

            Então para ligarmos dois hosts sem o auxilio de um hub ou um switch, utilizaremos um cabo cross.

        É importante deixar claro que é o cruzamento dos cabos que permite a comunicação sem a colisão de dados, ou seja, os dois lados no padrão T568B iria ocasionar a colisão
        do mesmo jeito.

        As placas de rede mais modernas já conseguem detectar esta diferenã e fazem a correção automaticamente via software. Mas nem todas as placas de rede possuem essa
        configuração. É melhor saber qual tipo de cabo precisamos usar.

        
    Questões aula 03:

        1 - Quais são os dois tipos de cabos que usamos para conexão?

            Selecione uma alternativa

            R: Cabo direto e cabo cruzado

            Caso tenhamos o mesmo padrão de cores nas duas pontas do cabo, chamamos de cabo direto, pois as mesmas cores estão nas mesmas posições nas duas pontas.
            Caso tenhamos um padrão de cores diferente em cada ponta do cabo, teremos o que chamamos de cabo cruzado.

        2 - O que caracteriza um cabo cruzado?

            Selecione uma alternativa

            R: Um cabo que possui os fios internos em posições diferentes em cada ponta

            Quando temos um padrão de cores diferente em cada ponta do cabo, teremos o que chamamos de cabo cruzado.

        
        3 - O que caracteriza um cabo direto?

            Selecione uma alternativa

            R: Um cabo com a mesma sequência de cor nas mesmas posições nas duas pontas do cabo

            Quando temos um padrão de cores iguais nas duas pontas do cabo, teremos o que chamamos de cabo direto

        
        4 - Quais são os dois padrões feitos pela TIA?

            Selecione uma alternativa

            R: T568A e T568B

            rede.

            O nome deste modo de operação é Broadcast. Imagine fazermos download de 500 mb através de um hub na rede e todos os outros dispositivos conectados a ele recebendo
            a todo momento esta informação....Isto causaria uma lentidão na rede.

            Isto nos leva a outra questão....Segurança da informação. Pelo nosso hub enviar a informação para todas as portas, em outras palavras, para todo mundo que estiver 
            conectado, podemos ter um usuário malicioso no terceiro computador, por exemplo, e ele pode fazer o que chamamos de análise de protocolo e decifrar o que está sendo 
            enviado pelo segundo computador. Além de lentidão, o hub representa vulnerabilidade da segurança.

        Para saber mais: Camadas de protocolos:

            Vimos que o protocolo TCP está acima da camada do protocolo IP. Os protocolos em redes seguem uma hierarquia e cada um é responsável por determinada função na
            comunicação.

            Antigamente cada fabricante de equipamento de redes desenvolvia protocolos proprietários tornando impossível a comunicação de rede com equipamentos feitos por outros
            fabricantes, criando assim o chamado "vendor lock-in".

            Desta forma foi criado um modelo de protocolo que faria com que qualquer hardware ou software pudesse se comunicar mesmo que um equipamento possuísse mais recursos
            que outro. Para isso foi definido que esses protocolos de comunicação seriam divididos em 7 camadas de comunicação, o chamado modelo OSI (Open System Interconnection).
            O protocolo TCP se encontra na camada 4, que é a camada de transporte. Já o protocolo IP se encontra na camada 3, que é a camada de rede.

            A parte de protocolos é muito extensa e não será abordada, porém é interessante estudar o modelo OSI.

Aula 05: Switches e seus avanços -------

    Teoria:

        Nós vimos que os hub's são limitados, não conseguem gravar qual é a porta de determinado computador mandando as informações para todas as portas gerando lentidão e
        falta de segurança na rede.

        Foi com a intenção de resolver as limitações do hub que criaram outros equipamentos de rede, um deles é o Switch.

        Criamos o mesmo ambiente de rede da outra aula no Packet Tracer, com uma única diferença, trocamos o hub pelo Switch.

        A forma do Switch trabalhar é parecida com o hub, ele recebe informações nas portas 1 e 2, e transmite informações nas portas 3 e 6.
        Utilizamos cabos diretos também para ligar os 3 computadores ao Switch.

        O Switch precisa de um tempo para habilitar a porta de comunicação após o cabo direto estar conectado. O processo demora alguns segundo e é retratado na simulação do 
        Packet tracer. A medida que as portas vão sendo habilitadas elas ficam verdes. 

        Mudamos o ambiente de realtime para simulation novamente, para que possamos verificar passo a passo o que está acontecendo.

        Realizamos o ping da mesma forma que anteriormente, do computador 2 para o computador 1.

        Aqui também o protocolo ICMP está dentro do ping e o ARP, que não sabe onde está localizado o IP, perguntara para todos na rede.

        Da mesma forma que o hub, o Switch irá disparar a informação para todos na primeira comunicação, com exceção da máquina de onde foi orifinada a informação.

        A máquina 3 descartará a informação pois seu IP é diferente daquele para o qual enviamos o pacote, mas o laptop responderá com outra informação.

        Nesta parte o hub continuaria enviando as informações para os dois computadores. Porém o Switch vai aprender quem é quem nesta primeira comunicação e vai registrar
        estas informações internamente. Nas próximas vezes que tentarmos enviar informações ele já saberá para quem enviar.

        Quando enviamos o protocolo ARP estamos perguntando quem tem determinado endereço IP, estamos na verdade pergutnando qual é o endereço mac com aquele número IP.

        Endereço Mac é o endereço físico de nossa máquina, ou mais objetivamente, da nossa placa de rede. Exemplo de endereço Mac: 81-E7-51-17-35-77
        Podemos verificá-lo com o comando: 
        
            > ipconfig /all

        Este é um número único que vem direto do fabricante.

        Uma vulnerabilidade do Switch que os hackers podem se aproveitar é de sua "tabela de memória". Se encher ela de endereços falsos, lotando sua memóriao aparelho
        já não poderá mais decifrar o endereço de cada computador e voltará a enviar informações para todas as máquina, atuando da mesma forma que um hub.

        O Switch trouxe grandes melhorias, mas ainda possui suas fragilidades que podem ser exploradas por hackers.

        Algo que podemos fazer com prevenção a este tipo de ataque é o que chamamos de segurança da porta. Em que cada porta tem o número máximo de endereços mac que pode 
        aceitar pré-definido. Caso haja um ataque, tentando enviar mais endereços mac para determinada porta, ela fica desabilitada.


    Questões aula 05:

        1 - O que é o protocolo ARP?

            Selecione uma alternativa

            R: Quando eu não conheço um dispositivo eu lanço esse protocolo para procurar o IP que eu quero me comunicar. A máquina buscada por sua vez vai retornar 
            informando seu endereço MAC

            O ARP é o protocolo utilizado para fazer o mapeamento entre o endereço IP e o endereço MAC de um dispositivo. Isso é necessário porque o MAC encontra-se 
            um nível abaixo do IP e eu preciso dele para poder transmitir as informações.

            Em redes de computadores, temos protocolos que possuem hierarquias diferentes. Para poder chegar até o IP que está na camada 3, eu preciso passar pelo MAC que está
            na camada 2, pense como se fosse escalar uma pirâmide, não dá pra chegar ao topo sem passar pelo meio dela!


        2 - Qual equipamento veio para substituir os hubs?

            Selecione uma alternativa

            R: Switches


        3 - Qual é a principal diferença entre os Hubs e os Switches?

            Selecione uma alternativa

            R: O Hub não consegue aprender onde um equipamento está localizado, o Switch sim

            Os hubs não conseguem aprender o endereço MAC das máquinas, já os Switches possuem essa função.


        4 - Como o Switch aprende onde um equipamento está localizado?

            Selecione uma alternativa

            R: Quando um dispositivo quer se comunicar com outro, ele vai necessitar passar pelo Switch e ele informa dentro do pacote qual é seu endereço MAC e o Switch grava 
            essa informação em sua memória.

            Os dispositivos ao se comunicarem passam pelo Switch e ao passar pelo Switch ele grava em sua memória quem está conectado em qual porta.

        
        5 - Qual é uma forma de ataque usada para conseguir informações passadas no Switch destinadas a outro usuário?

            Selecione uma alternativa

            Métodos usados por usuários maliciosos seria de inserir vários endereços MAC falsos para “lotar” a memória do Switch, uma vez que a memória esteja cheia, o Switch

            Podemos configurar a porta do Switch para aceitar um número máximo de endereços MAC, ao ultrapassar esse limite a porta é desligada e o ataque não teria sucesso.

        
Aula 06: Roteadores e a comunicação externa -------

    Teoria:

        Máscaras de rede:

            Mesmo que os computadores estejam conectados no mesmo switch ou no mesmo hub, não significa que eles estão na mesma rede.

            Nós podemos fazer uma separação lógica de endereçamento e colocar cada computador em uma rede diferente.
            Precisamos as vezes segmentar uma rede em duas ou mais para atender vários tipos de necessidades. Separar setores de uma mesma empresa, por exemplo.

            Primeiro ponto a se considerar é: como saberemos se o computador está na mesma rede?

            Quando entramos nas propriedades da nossa conexão de rede para configurar o nosso número IP de forma manual, vemos um campo chamado: "Subnet Mask" ou Máscara de 
            sub-rede.

            No caso do laboratório que utilizamos, deixamos este campo configurado com 255.255.255.0
            Esta é uma informação importante para saber se os computadores estão na mesma rede.
            Façamos uma analise...

                Endereço IP: 192.168.3.2
                Máscara de rede: 255.255.255.0

            A máscara de rede dividi o endereço IP do computador em dois grandes grupos: um referente às redes e outro às máquinas.

                255 = Rede
                0 = Máquina (Host)

            Tudo o que for 255 será referente à rede, e o que for 0 será referente ao host. Então, ele analisará o primeiro intervalo:

            Então, definimos que, para outro dispositivo estar na mesma rede do nosso computador configurado logo acima, ele precisará começar com o mesmo intervalo de octeto.
            Logo, ele deverá começar por 192.

            Então ele passará para o outro intervalo: Se este dispositivo quiser estar na mesma rede, deverá ter o segundo octeto 168. E o mesmo no terceiro octeto: O computador
            deverá ser igual a 3.

            Ou seja, sobrou só o ultimo octeto para o número de máquinas. Então para qualquer outro computador esta na mesma rede do computador acima, terá que ter o início:

                192.168.3

            O ultimo octeto fica livre, e qualquer numero que outros pc's configurem neste grupo, conseguirá se comunicar com nosso computador.

            Exemplo: 192.168.3.1

                        ou

                     192.168.3.3

                     ou até mesmo

                     192.168.3.77

            O que realmente importa é que para estar na mesma rede o endereço IP deve começar por 192.168.3.

            Por isso que no nosso laboratório todas as máquinas foram configuradas com o ínicio 192.168.3. Qualquer um dos três intervalos que tivessem sido mudados em nossa
            configuração e qualquer uma das máquinas, iria impedir esta mesma máquina de se comunicar na rede.

            O teste realizado nesta aula foi mudar o primeiro intervalo para 191, deixando o IP da primeira máquina assim:

                191.168.3.1

            Ao realizar novamente o teste de ping da máquina 2 para a máquina 1 ele não conseguiu enviar os pacotes perdendo todos os 4.
            A conexão não foi estabelecida e a máquina não respondeu.
            Ou seja, com o Hub e com o Switch só conseguimos nos conectar com computadores que estão na mesma rede.

        
        Roteadores:

            Para fazer com que dois computadores de redes diferentes se comuniquem, precisaremos de um roteador.

            Nesta aula deletamos o Switch e um dos computadores (3), e adicionamos um Router e deixamos os outros dois computadores.

            Primeiro verificamos se estamos conectando equipamentos com placas de rede semelhantes, como estamos conectando os dois computadores ao Router, não estamos conectando
            equipamentos semelhantes.

            Neste exemplo o Router não esta sendo usado de forma eficiente pois estamos conectando apenas dois computadores a ele, enquanto ele poderia conectar redes inteiras de 
            setores diferentes.

            Por isso, utilizamos cabo cruzado no projeto desta aula. Pois o roteador não estará conectando redes diferentes, mas sim, apenas duas máquinas.

            Conectamos a porta fast ethernet dos dois computadores as portas disponíveis do roteador.

            Criamos também a segmentação por setores da empresa, de ínicio, financeiro e marketing.          
            

        Configurando roteador:

            Vimos uma imagem de um roteador doméstico no curso, com 4 portas amarelas e 1 porta azul.
            As portas amarelas são do "Switch" embutido no aparelho. Isto acontece nos equipamentos domésticos para economizar espaço.

            Os equipamentos que temos no simulador Packet Tracer da Cisco são focados para empresa. E em uma empresa estes equipamentos (Switch e Router) ficam separados.
            Switch para conectar computadores e usuários na mesma rede, Roteador para conectar redes.

            Por isso só teremos duas portas no roteador do projeto que estamos fazendo.

            Para conseguirmos realizar a configuração do Roteador precisaremos de um cabo chamado, cabo console.
            Ele é um cabo azul, que em uma das pontas é RJ45 e na outra ponta o conector Rs232, referente a uma porta serial (que transmite os bits um por vez).

            O RJ-45 é conectado na porta do roteador e o Rs232 em um computador onde iremos definir as configurações. Porém, a maioria dos computadores não vem mais com a 
            porta serial, então precisamos usar um adaptador.

            Um programa que podemos utilizar para realizar a configuração do equipamento da Cisco é o PuTTy.

            O PuTTY é muito útil quando queremos acessar remotamente um equipamento. Podemos remotamente configurar um aparelho. Caso tenha alguém na outra cidade para nos
            auxiliar com o atendimento remoto não precisaremos nos deslocar para realizar a configuração. A interface gráfica dele é amigável.

            No nosso laboratório desta aula, mantemos dois computadores e um roteador, conectados por cabo cross over.
            Adicionamos apenas um computador com a conexão do cabo Rs232 na porta console do roteador para simular a configuração do roteador.

            Porém isto é na prática, já no nosso simulador da Cisco as coisas são um pouco mais simples, pois podemos clicar no roteaodr e ir até a aba CLI 
            (Command Line Interface).

            Na prática precisaríamos fazer a conexão via computador. Para isto clicaremos no computador e utilizaremos o terminal.

            No terminal nos será perguntado como queremos conectar com o roteador. Nós deixaremos as velocidades padrões. A partir de agora, faremos as configurações clicando
            diretamente no roteador. Se quisermos podemos fazer a configuração do roteador clicando no computador do simulador.

Aula 07: Entendendo os endereços IP -------

    Teoria:

        Classes IP:

            Precisamos configurar IP's nas máquinas para que elas se comuniquem, mas não podemos criar qualquer valor em qualquer intervalod de endereço. Por isso existe um 
            órgão internacional que regulamenta os IP's colocando em classes que podem ser utilizadas.

            Classe A:

                Classe de endereçamento: A
                Intervalo 1º octeto: 1 - 126
                Máscara de rede: 255.0.0.0

                Para que um endereço IP esteja dentro da classe A, o primeiro octeto tem que estar entre 1-126 e a máscara de rede padrão 255.0.0.0

                O que for 255 é parte da rede e o que for 0 estará reservado para os host's.

                A máscara faz a análise o IP e para cada octeto seu que for 255 ela verificará no respectivo octeto do IP é igual ao outro ip que esta sendo comparado.

                Ex:

                Se a máscara for 255.0.0.0 e o IP da máquina for 124.4.7.9 o IP da outra máquina na mesma rede deverá ter obrigatoriamente o primeiro octeto igual, que é: 124

            Classe B:

                Classe de endereçamento: B
                Intervalo do 1º octeto: 128-191
                Máscara de rede: 255.255.0.0

                Neste caso da Classe B, como a máscara é 255.255.0.0 os dois primeiros octetos dos IP's que estiverem na mesma rede tem que ser igual.

                Ex:

                    172.161.X.X, então os dois octetos seguintes são reservados para host, não importante seu valor para que os computadores se comuniquem.

            
            Classe C:

                Classe de endereçamento: C
                Intervalo do 1º octeto: 192-223
                Máscara de rede: 255.255.255.0

                Nesta classe o primeiro Octeto deve estar entre 192 e 223 e a máscara padrão é 255.255.255.0, assim como nosso laboratório funcionou.

                Na classe C os dois octetos seguintes, segundo e terceiro, tem que ser o mesmo para IP's que ficarão na mesma rede.

            
            Classe D: 

                Classe de endereçamento: D
                Intervalo do 1º octeto: 224-239
                Máscara de rede: -------

                Primeiro octeto nesta classe tem que estar entre 224-239.

                É uma classe reservada para multicast (caso em que queremos fazer a comunicação somente com alguns dispositivos que estão na nossa rede).
                Ela não é atribuida para máquinas.

            Classe E:

                Classe de endereçamento: E
                Intervalo do 1º octeto: 240-255
                Máscara de rede: -------

                Assim como a classe D esta classe também é reservada, não sendo utilizada para atribuir endereços IP's para máquinas.


        IP privado:

            Dentro de cada uma das classes de IP, teremos faixas que são chamadas privadas. Elas recebem este nome porque só conseguem se comunicar na rede local, não podendo
            ser utilizados na comunicação na internet.

            Intervalo para endereços IP privados na classe de endereços A:

                10.X.X.X

                Todo o IP da rede que começar com 10, não poderá se comunicar com a internet diretamente.


            Intervalo para IP's privados de classe B:

                172.16.X.X a 172.31.X.X

            Intervalo para IP's privados de classe C:

                192.168.X.X

            No computador de nossa casa podemos utilizar o comando "ipconfig" no terminal e verificar qual é nosso endereço IP, em que classe está e se é privado ou não.
            Se for privado, como se conecta com a internet?

                Conseguimos porque o nosso roteador traduz o nosso IP privado para um IP público usando o NAT (Network Address Translation) em que é feita a tradução de um
                endereço privado para o público.


        IPv6:

            Se analisarmos com o comando "ipconfig" o nosso endereço IP, veremos que se trata de um protocolo IPv4.

            O v4, que significa "versão 4", chegou ao fim devido a grande popularidade da internet. Por isso tiveram que criar o IPv6, que possui uma série de novos componentes
            e funcionalidades mais avançadas.

                2804:49c:3102:401:ffff:ffff:ffff:36

            O site do UOL no caso, ainda possuí um endereço IPv4 (200.147.3.157) para atender diferentes tipos de acesso que tenhamos na máquina deles.
            Podemos perceber que a versão 6 possui muito mais informações que a versão 4. Lembrando que atualmente o IPv4 não possui endereços IP's disponíveis.
            O IPv6 já é o padrão utilizado.

        
        Endereços reservados classe A:

            Temos um range de IP para utilizarmos? Qual seria este range?

            Como um exemplo de endereço que não podemos utilizar temos: 2000.2.4.5. Isto porque nem um dos 4 octetos podem passar de 255.

            Existe tanto um número mínimo e máximo que um endereço IP pode ter. No caso, o mínimo é 0 e o máximo é 255. Estes números, porém, não podem estar em qualquer ordem.
            Todo endereço IP deve estar inserido dentro de uma rede pré-definida, que terá um número de identificação e ainda, terá o endereço de rede e o endereço que serve 
            para comunicar com todos os dispositivos da rede chamado broadcast.

            Exemplo de como descobrimos qual é o endereço de rede e o de broadcast:

                Endereço IP: 124.4.7.9

                Máscara de rede: 255.0.0.0

                Lembrando que a quantidade de redes é representada pelos 255 e os hosts pelo 0.

                Como nosso primeiro octeto do endereço IP está entre 1 e 126, se trata de um endereço IP de classe A. Por isso a máscara de rede é: 255.0.0.0.

                Para descobrirmos o endereço de rede cortaremos o 255 da máscara de rede.

                    255.0.0.0

                        .0.0.0

                Verificamos que ficaram os 3 zeros dos octetos seguintes.

                Comparamos com o nosso endereço IP.

                    124.4.7.9
                        .0.0.0

                Basta então mover os grupos de cima que ficaram vazios para completar os octetos que foram recortados embaixo.
                Ficará assim:

                    124.0.0.0 = Este é o endereço de rede!

                Agora para descobrirmos o endereço de broadcast faremos o trabalho ao contrário. Cortaremos os 0 do endereço de rede.
                Ficaremos assim:

                    124...

                Agora adicionamos o 255 nos espaços recortados. Teremos o seguinte resultado:

                    124.255.255.255

                Este então, é o endereço de broadcast: 124.255.255.255
                Este endereço também não poderá ser usado em nenhuma máquina.

                Criamos um novo projeto no Packet Tracer colocando dois micros e um switch.
                O primeiro micro com IP 124.0.0.255 e o segundo micro com IP 124.1.0.0.

                Os dois endereços IP's são válidos e irão se comunicar, podemos realizar o teste com o protocolo ping.
                Isso porque o padrão está correto, IP's de classe A com máscara 255.0.0.0 precisam obrigatoriamente ter apenas o primeiro grupo igual.

        
        Endereços reservados classe B:

            Seguindo o mesmo exemplo anterior, vamos descobrir qual é o endereço de rede e o endereço de broadcast do IP proposto:

                Endereço IP: 172.161.7.9

                Máscara de rede: 255.255.0.0

                255 =  Rede
                0 = Host

                Sabendo que em endereços de classe B o primeiro octeto vai de 128 a 191, percebemos que o nosso endereço IP logo acima é de classe B.

                Recortamos os 255's da mascara e substituimos pelos os valores dos octetos correspondentes:

                    255.255.0.0 fica ..0.0

                    Então, substituindo valores:

                    172.161.0.0 = Este é o nosso endereço de rede.

                    Agora para descobrirmos o broadcast substituimos os zeros por 255, teremos o seguinte valor:

                    172.161.255.255 = Endereço de broadcast

                    Lambrando que os endereços 172.161.0.0 e 172.161.255.255 não podem ser atribuidos a nenhuma outra máquina na rede.

                    No laboratório criamos mais um ambiente, mais duas máquina e mais um switch.
                    A primeira máquina com endereço 172.161.0.255 e a segunda máquina com endereço 172.161.1.0.
                    Realizamos testes de ping novamente.

        
        Endereços reservados classe C:

            Vamos ver qual é o endereço de rede e o endereço de broadcast dos IP's da classe C:

                O IP que nos foi dado é o seguinte:

                    Endereço IP: 192.168.0.9

                    Máscara de rede: 255.255.255.0

                    255 = Rede
                    0 = Máquina (Host)

                    O intervalo da classe C se dá entre 192-223, como nosso endereço IP acima começa com 192 já sabemos a qual classe ele pertence.

                    A máscara padrão da classe C é 255.255.255.0, sendo assim os 3 primeiros grupos do nosso IP devem ser presevados caso outro computador queira estar na mesma rede e 
                    para que possamos encontrar o endereço de rede e o de Broadcast.

                    Repetindo o procedimento para que possamos encontrar o endereço de rede, recortaremos os três primeiros octetos da máscara e adicionaremos os respectivos grupos do
                    nosso IP, que no caso é:

                        Para: 255.255.255.0

                        deixamos: ...0

                        Completamos os octetos que ficaram vazios: 192.168.0.0 (Este é o endereço de rede)

                    Agora encontraremos o endereço de broadcast.

                        Devemos agora remover os 0's adicionados no procedimento para encontrar o endereço de rede. Como adicionamos apenas 1 zero, removeremos apenas seu octeto.

                        Ficando: 192.168.0.

                        Agora neste octeto que ficou vazio, iremos adicionar o 255.

                        Então: 192.168.0.255 (Endereço de broadcast)    

                Porém não podemos nos prender a ideia de que não podemos ter o final de um IP com 0 ou 255. O que definirá serão os números de endereço de rede e de broadcast.

                                     
    Questões aula 07:

        1 - Quais são as classes de endereços IP que podem ser endereçadas para máquinas?

            Selecione uma alternativa

            R: Classe A,B e C

            A IETF (Internet Engineering Task Force) determinou que existiriam ao todo 5 classes de endereços IP, indo de ordem alfabética da classe A até a classe E. 
            Porém as duas últimas classes não são usadas para serem endereçadas as máquinas. A classe D seria usada para multicast (termo usado quando queremos nos comunicar 
            com somente algumas máquinas de nossa rede) e a classe E seria uma classe experimental. Portanto as classes de IP que podem ser endereçadas para máquinas seriam a 
            classe A, B e C.


        2 - Como eu identifico que um endereço IP está em uma classe?

            Selecione uma alternativa

            R: Pelo valor do primeiro intervalo (octeto)

            Para sabermos em qual classe um endereço IP se encontra, temos que analisar o primeiro octeto e ver dentro de qual range ele estaria. (Classe A, B ou C).


        3 - O endereço IP 187.77.45.8 estaria dentro de qual classe, considerando que usa máscara de rede padrão?

            Selecione uma alternativa

            R: Classe B. 
            
            A classe "A" possui o primeiro octeto variando de 1 a 127, a classe "B" possui o primeiro octeto variando de 128 a 191 e a classe "C" possui o primeiro octeto 
            variando de 192 a 223. Dessa forma pelo fato do número 187 se encontrar dentro de 128 a 191, sabemos que é um endereço da classe "B".

        
        4 - O que seria o endereço 127.0.0.1? Como ele é conhecido?

            Selecione uma alternativa

            R: É um endereço interno da placa de rede, usado para testar se os protocolos TCP/IP estão funcionando. Ele é conhecido como endereço de loopback, pois o sinal
            é enviado e recebido por ele mesmo.

            O endereço 127.0.0.1 seria uma faixa de endereço IP reservada, esse seria um endereço interno da placa de rede para realizar testes e verificar se ela está de 
            fato validando os protocolos TCP/IP.

        
        5 - O que são IPs privados?

            Selecione uma alternativa

            R: São IPs que são usados somente para comunicação na rede local, não posso acessar a internet

            Os endereços IP privados são usados para comunicação somente em minha rede local, de acordo com
            a especificação, eles não podem ser usados para comunicação na internet por exemplo.

        
        6 - Se eu tenho IP privado na minha máquina, como posso acessar a internet?

            Selecione uma alternativa

            R: Através do método de tradução de endereços IPs privados para públicos, chamado de NAT

            Isso acontece porque nosso roteador possui a configuração chamada NAT, essa configuração
            vai converter o endereço IP privado que temos em nossas máquinas para IP públicos que nosso provedor de serviços nos fornece.

        7 - Qual foi a evolução do IPv4?

            Selecione uma alternativa

            R: IPv6

            A evolução do IPv4 seria a versão 6 do protocolo, que seria conhecida como IPv6.

        
        8 - Por que foi necessário o IPv6?

            Selecione uma alternativa

            R: Porque os endereços públicos do IPv4 chegaram a um fim

            O endereço IPv6 foi necessário porque os endereços IPv4 públicos chegaram a um fim por conta da grande popularidade da internet, smartphones, tablets, etc.

        
        9 - Dado que o endereço IP da máquina 7.8.7.8 possui máscara de rede 255.0.0.0, determine seu endereço de rede e broadcast (Lembre-se da regra, endereço de rede e 
            broadcast):

            Selecione uma alternativa

            R: Rede: 7.0.0.0 ; Broadcast: 7.255.255.255

            Descobrir endereço de rede que esse endereço IP está inserido:
            Se recortarmos o 255 da máscara de rede e inserirmos o octeto correspondente do endereço IP, teremos: 7.0.0.0 :)
            
            Descobrir o endereço de broadcast da rede:
            Pegamos o endereço de rede e recortamos os 0’s (originais da máscara) e colocamos 255 no lugar, teremos então: 7.255.255.255
            Dessa forma, por exemplo o endereço IP: 7.0.0.255 é válido porque é maior que o endereço de rede (7.0.0.0) e menor que o de broadcast (7.255.255.255)

        
        10 - Dado que o endereço IP da máquina 135.44.3.21 possui máscara de rede 255.255.0.0, determine seu endereço de rede e broadcast (Lembre-se da regra, endereço de rede
            e broadcast):

            Selecione uma alternativa

            R: Rede: 135.44.0.0 ; Broadcast: 135.44.255.255

            Descobrir endereço de rede que esse endereço IP está inserido:
                Se recortarmos o 255 da máscara de rede e inserirmos os octetos correspondentes do endereço IP, teremos: 135.44.0.0 :)
            
            Descobrir o endereço de broadcast da rede:
                Pegamos o endereço de rede e recortamos os 0’s (originais da máscara) e colocamos 255 no lugar, teremos então: 135.44.255.255

            Dessa forma, por exemplo o endereço IP: 135.44.0.255 é válido por que é maior que o endereço de rede (135.44.0.0) e menor que o de broadcast (135.44.255.255)

        
        11 - O que caracteriza uma comunicação broadcast?

            Selecione uma alternativa

Aula 08: Endereçamento IP com DHCP -------

    Teoria:

        DHCP:

            Até o momento temos feito a configuração manual de todos os endereços IP.

            Numa empresa com 40, 50 ou mais computadores isto se tornaria muito trabalhoso.

            Uma forma de resolver este problema de forma automática é o servidor DHCP.

            Criamos um projeto com 4 computadores ligados a um Switch e este Switch ligado a um Router.

            O roteador da Cisco que irá atuar como servidor5 DHCP.

            Ele fornecerá automaticamente endereços IP's para os computadores.

            Vemos então neste cenário um pequeno modelo de rede natural onde o Switch conecta vários computadores (Conexão de micros na rede local) e o nosso roteador está conectado ao
            Switch (Conectando a rede). Lembrando que temos outras portas no roteador que poderia conectar outras redes simultaneamente.

            Para configurar o Roteador conectamos ele a um notebook utilizando o cabo console. Como vimos antes o Packet Tracer nos permite usar o CLI diretamente no roteador não sendo
            necessária a conexão com o notebook.

            Configurando o roteador:

                Primeira coisa a se fazer é habilitar a porta.

                O CLI nos perguntará se queremos o diálogo para configurar o roteador, digitamos "no".

                Após apertar "return" para continuar com a operação estaremos no modo usuário que apenas nos permite verificar algumas coisas, como reparos e testes, mas não podemos
                configurar o equipamento. Teremos que aumentar os privilégios:

                    Router>enable

                    Saída:

                        Router#

                Até aqui só poderemos ver todas as visualizações que poderemos fazer no equipamento. Seguiremos para o configure terminal:

                    Router#configure terminal

                    Saída:
                        
                        Enter configuration commands, one per line.  End with CNTL/Z.
                        Router(config)#

                Agora estamos efetivamente na parte de configurações.

                Vamos iniciar efetivamente a configuração da porta na qual conectamos o Switch.

                    Router(config)#interface GigabitEthernet 0/0/0
                    
                    Saída:

                        Router(config-if)#


                Iremos inserir o comando para ligar a porta:

                Router(config-if)#no shutdown

                Saída:

                    Router(config-if)#
                    %LINK-5-CHANGED: Interface GigabitEthernet0/0/0, changed state to up

                    %LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/0/0, changed state to up

                Agora as portas estão habilitadas e as luzes devem ficar verdes. O CLI nos informa que o estado da porta foi mudado para "up". A porta já está ativa.

                Iremos então começar a configuração para que ele trabalhe com DHCP. Primeiro temos que sair da configuração que estamos:

                Router(config-if)#exit
                
                Sáida:
                
                    Router(config)#

                Utilizamos o comando:

                    Router(config)#ip dhcp

                    Porém somos informados que o comando está incompleto.

                    Precisamos criar um pool de endereços para que o nosso servidor DHCP utilize, aqueles que serão atribuídos para os computadores. Daremos o nome de cursoRedes para
                    esse pool.

                        Router(config)#ip dhcp pool cursoRedes

                        Saída:

                            Router(dhcp-config)#

                    Iremos inserir "?" e digitaremos enter para ver quais comando faltam (ou parâmetros) para terminar de configurar nosso servidor DHCP.

                        Router(dhcp-config)#?

                        Saída:

                            default-router  Default routers
                            dns-server      Set name server
                            domain-name     Domain name
                            exit            Exit from DHCP pool configuration mode
                            network         Network number and mask
                            no              Negate a command or set its defaults
                            option          Raw DHCP options

                    Utilizaremos o comando "network" para informar a rede que queremos trabalhar.

                        Router(dhcp-config)#network 192.168.0.0

                        Seremos informados que o comando está incompleto:

                            % Incomplete command.

                    Precisamos verificar quais parâmetros faltam para que possamos terminar de configurar.

                    Router(dhcp-config)#network 192.168.0.0 ?
                    
                    Saída:

                        A.B.C.D  Network mask

                    Ficou faltando a máscara de rede, que será: 255.255.255.0

                    Então nosso comando completo ficará:

                        Router(dhcp-config)#network 192.168.0.0 255.255.255.0

                        Se quisermos podemos adicionar o "?" no comando acima para ver se falta algo antes de apertar "Enter".

                            Router(dhcp-config)#network 192.168.0.0 255.255.255.0 ?

                            Saída:

                                <cr> 

                                <cr> = Signífica "Caracter return", que não existe mais nada a ser adicionado a este comando.

                        Já informamos qual é a rede que o pool de conexões estará trabalhando.

                    Será importante indicar qual será o gateway para os computadores, ou seja, para onde as máquinas devem enviar informações caso eles precisarem se comunicar com alguém
                    externamente. Utilizaremos o comando:

                        Router(dhcp-config)#default-router ?
                        
                        Saída:
                        
                            A.B.C.D  Router's IP address

                            Veremos que falta apenas o endereço IP do default router, que é o endereço da porta que estamos utilizando no roteador. Utilizaremos o IP: 192.168.0.1.

                                Router(dhcp-config)#default-router 192.168.0.1

                                    Obs: Mesmo que já tenhamos definido qual é o endereço IP e a porta do Roteador que estamos utilizando, é necessário o comando "default-router" para
                                    mostrar qual é a porta do Gateway e assim fazendo que ela seja preenchida automaticamente na distribuição de IP via DHCP.

                            Lembrando que pelo fato de estarmos utilizando a máscara: 255.255.255.0 temos que utilizar os 3 primeiros octetos que definimos.

                                Router(config)#interface gigabitEthernet 0/0/0

                                Saída:

                                    Router(config-if)#

                            Agora iremos inserir o IP que utilizaremos nesta porta:

                                Router(config-if)#ip address 192.168.0.1 255.255.255.0

                                Lembrando que podemos utilizar o "?" todas vez que acharmos que falta um parâmetro do comando.

                    Vamos verificar se a configuração DHCP funcionou. Colocaremos todos os computadores para receber o IP de forma DHCP.

                    Podemos observar que a partir do momento que mudamos a configuraçãos dos Host's para DHCP eles fazem uma requisição de IP para o servidor DHCP do servidor.

                    O procedimento de requisição e reposta é o seguinte. No primeiro momento ele não tem IP algum e saíra perguntando quem irá fornece-lo.
                    Este é um final broadcast. O roteador que configuramos irá receber a requisição e dirá: "Estou configurado para oferecer endereços IP's".O computador aceitará o IP
                    fornecido e irá informar que recebeu um IP. A mensagem será entregue para o roteador também, que por sua vez, irá dar um "OK".

                    O processo é o mesmo para qualquer máquina que for conectada a rede, mudamos em sua placa de rede de "Static IP" para "DHCP". Tanto o IP é configurado automaticamente
                    quanto a máscara e o gatway.

                    Quando finalizamos os 4 computadores receberam IP de forma automática e dinâmica sem que conflitem. Não precisamos mais configurá-los manualmente.

    
    Questões aula 08:

        1 - Para que serve um servidor DHCP?

            Selecione uma alternativa

            R: Para atribuir IP a máquinas (clientes) de forma dinâmica

            Os servidores DHCP (Dynamic Host Configuration Protocol) alocam dinamicamente endereços IPs a clientes (máquinas).

        
        2 - Como é conhecida essa forma de atribuição de IP pelo DHCP?

            Selecione uma alternativa

            R: IP dinâmico

            Quando um endereço IP é atribuído a uma máquina (cliente), dizemos que a configuração foi dinamicamente alocado. Os servidores DHCP normalmente possuem o que chamamos de
            “lease time”, ou seja possui um tempo de alocação de um endereço IP a uma máquina, quando esse tempo é expirado é preciso ocorrer uma renovação de endereço IP. Por isso ele 
            é dinamicamente alocado :)

        
        3 - Quando um cliente não possui um endereço IP e está configurado para receber IP dinâmico, como ele faz a requisição para que alguém forneça um endereço IP?

            Selecione uma alternativa

            R: Broadcast

            Quando um cliente não possui endereço IP ele não sabe a quem perguntar, então ele precisa sair perguntando para todo mundo que está na mesma rede quem poderá fornecer um 
            endereço IP. Quando essa comunicação é feita para todos os dispositivos, chamamos isso de Broadcast.


Aula 09: Acessando o site em nossa simulação -------

    Teoria:

        Acessando o Google:

            Agora nossa missão é permitir que os usuários acessem o site do Google ou pelo menos uma simulação disto.
            Devemos lembrar que estamos usando o software de simulação da Cisco então eles não acessarão a Web de verdade.
            Configuramos o servidor e uma página Http: para simular esta conexão.
            Estamos usando no projeto 4 PC's, 1 Switch e 1 Router da aula 8.
            Acicionaremos mais 1 Switch e 1 Server e conectaremos ao nosso roteador DHCP.

            Assim como tivemos que ativar a porta do Router GigabitEthernet 0/0/0 teremos que ativar a GigabitEthernet 0/0/1 para se comunicar com o Switch que sai para a Web até o servidor.

            Voltando para o CLI do nosso roteador utilizamos os seguintes comandos em sequência para adiantar o trabalho:

                Router>enable //Elevando permissões para executarmos comandos

                Router#configure terminal //Entrando em modo de configuração

                Saída:
                    
                    Enter configuration commands, one per line.  End with CNTL/Z.
                
                Router(config)#interface gigabitethernet 0/0/1 //Escolhendo a porta que iremos configurar

                Router(config-if)#no shutdown //Ativando a porta

                Saída:

                    Router(config-if)#
                    %LINK-5-CHANGED: Interface GigabitEthernet0/0/1, changed state to up

                    '%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/0/1, changed state to up

            Já está escolhida a porta que configuraremos para a saída para a internet e a ativamos.

            Definiremos agora um endereço IP para esta porta:

                Router(config-if)#ip address ? //Para verificar o que precisamos para configurar o IP
                
                Saída:

                    A.B.C.D  IP address
                    dhcp     IP Address negotiated via DHCP

                    Logo acima a informação de que precisamos de um endereço IP e uma máscara de subrede.

            Definimos então o que é necessário:

                Router(config-if)#ip address 8.8.8.1 255.0.0.0

                Lembrando que, como escolhemos um endereço IP de classe A, nossa máscara de subrede por padrão também deve ser de classe A.

            Feita a configuração no nosso Router, iremos agora configurar o nosso servidor para realizar a simulação.
            Definiremos qualquer IP que esteja entre 8.0.0.0 e 8.255.255.255, pois será válido.

                No software Packet Tracer, no nosso Servidor Web definiremos:

                    Endereço IP: 8.0.255.255
                    Máscara de subrede: 255.0.0.0
                    Default Gateway: 8.8.8.1

            Editaremos uma página para representar a página que queremos acessar. Não precisamos de muito design pois é apenas uma simulação.
            No nosso servidor web iremos até services e depois em HTTP.

                Colocaremos o seguinte conteúdo como index.html:

                    <html>
                        <h1>GOOGLE</h1>
                        <input type="search">
                        <input type="button" value="Buscar">
                    </html>  

            Agora temos configurado o nosso roteador para atuar como DHCP para a nossa rede privada e um IP estático.

            Agora as duas redes já sabem qual é o portão de saída e podem se comunicar entre si. Faremos um teste.

            Podemos ver que o IP definido para o nosso roteador é o mesmo gateway do nosso servidor Web.

            Digitaremos no Web Browser de um dos computadores da nossa rede privada o IP do servidor do "Google" que criamos.

                http://8.0.255.255

            No próximo passo aprenderemos a configurar o servidor DNS para que ao invés de colocarmos o endereço IP na barra de endereços do browser, digitemos apenas a URL que será traduzida
            para o endereço IP.


        Configurando DNS:

            Vamos colocar o servidor DNS para fazer a tradução entre a URL www.google.com e o endereço IP máquina do Google. Adicionaremos o servidor DNS ao projeto

            Conectamos o novo servidor (DNS) ao Switch, aquele mesmo que o nosso servidor web já está conectado.

            Assim como nosso servidor web está configurado, também configuraremos o nosso servidor DNS com um endereço IP de classe A:

                Endereço IP: 8.1.0.0
                Máscara de subrede: 255.0.0.0
                Default Gateway: 8.8.8.1 (Endereço da porta de saída do nosso roteador)
                
            Agora vamos criar o mapeamento do nosso servidor DNS para o servidor Web.

            Em "Services" -> "DNS" digitaremos a URL do site que para qual o endereço IP do servidor web nos envia (www.google.com) e relacionaremos ao endereço IP deste servidor web
            (8.0.255.255).

            Habilitamos o servidor DNS clicando em "On" e depois em adicionar.

            Fazemos um teste indo até um dos micros e digitando no Web Browser o endereço que acabamos de mapear no servidor DNS.

            Porém pordemos perceber que não deu certo, isto porque não indicamos qual é o servidor DNS no nosso roteador. O computador não sabe qual é o servidor DNS que ele deve procurar.
            Como estamos usando o DHCP, pediremos que ele informe aos usuário squal o servidor DNS será utilizado.

            Voltamos a área CLI do nosso roteador.

                Router>enable //Elevando permissões

                Router#configure terminal //Entrando no modo de configuração
                Enter configuration commands, one per line.  End with CNTL/Z.

                Router(config)#ip dhcp pool cursoRedes //Entrando nas configurações de DHCP do nosso servidor DHCP

                Router(dhcp-config)#? //Olhando lista de comandos para verificar o comando daquilo que queremos configurar.
                default-router  Default routers
                dns-server      Set name server
                domain-name     Domain name
                exit            Exit from DHCP pool configuration mode
                network         Network number and mask
                no              Negate a command or set its defaults
                option          Raw DHCP options

                Router(dhcp-config)#dns-server 8.1.0.0 //Informando ao servidor DHCP o endereço IP do nosso servidor DNS

                Router(dhcp-config)#

            Como a distribuição de IP e as configurações já foram realizadas ao ligar a nossa rede, devemos realizar um "refresh" nos micros.
            Basta clicar na opção "Static IP" e voltar para "DHCP" ou realizar o comando:

                ipconfig /renew

            Agora nossos computadores sabem qual é o servidor DNS.
            
            Fazemos um novo teste digitando o endereço www.google.com no web browser de qualquer computador que realizamos o refresh.
            Verificamos agora que conseguimos visualizar a página do Google.

            Projeto finalizado!!!
